# UI 质量完善闭环

这个项目的界面优化不能只停留在单次调样式。后续每次调整工作台、画布、属性栏或底部状态层，都按“自提意见 -> 自审核 -> 自强化”的闭环执行，并用脚本把关键判断固化下来。

## 自提意见

- 把界面拆成六层：全局导航层、任务控制层、功能导航层、核心工作层、属性面板层、状态栏层。
- 核心工作层优先服务故障建模画布，仿真参数和运行按钮应放在任务控制层，避免压缩画布。
- 左侧组件库和右侧属性栏都需要贯穿到下方，形成连续的双侧工作面板。
- 底部状态层要更小、更紧凑、更协调，重点承载日志、告警、仿真结果、性能趋势和少量指标。
- 所有比例问题优先通过稳定栅格和可拖拽尺寸适配解决，而不是为单一截图硬编码。

## 自审核

运行：

```bash
npm run audit:ui
```

审查脚本会检查以下阻断项：

- 根布局是否保留全局导航、任务控制、主工作区三段高度。
- 仿真控制条是否已经从画布内部移动到任务控制层。
- 左侧组件库和右侧属性栏是否都跨过底部区域并贯穿到底。
- 底部状态层是否只位于中间画布下方。
- 左侧、右侧和底部尺寸是否支持拖动并持久化。
- 底部状态层是否有紧凑日志头、操作按钮和 2x2 指标卡网格。
- 底部日志操作是否具备真实行为：清空、级别过滤、导出和运行时事件追加。
- 底部指标卡是否采用数字、单位、分隔符的结构化排版，而不是整句粗体文本。
- 模型检查是否把错误、警告或通过结果写入底部日志/告警流。
- 画布工具栏是否都暴露明确命令：撤销、重做、视图切换、选择/平移、缩放、全屏和模型检查。
- 结构性画布操作是否进入可撤销历史，而不是一次性破坏状态。
- 是否移除了导致下方组件无法完整显示的旧固定高度限制。
- 是否保留键盘焦点和减少动态效果的基础可访问性规则。

## 自强化

- 新发现的 UI 比例、截断、错位问题，先转成 `tools/ui_quality_audit.mjs` 的阻断检查。
- 需要 DOM 结构验证的问题，同时补到 Vitest，例如：

```bash
npm test -- --run tests/ui-quality-audit.spec.js
```

- 需要视觉比例验证的问题，后续补充 Playwright 截图回归，覆盖 1440px、1536px、短高度屏幕和宽屏。
- 不确定是否立即修改的问题，先进入审查脚本的建议项，避免丢失设计债务。

当前底部状态层的质量目标是：小而完整、信息密度高、和日志表协调，不再出现组件不能完全显示的问题。

当前已吸收的强化项：

- 底部状态栏支持日志清空、级别过滤、导出 CSV。
- 运行时可通过 `window.__GZ_STATUS_BAR__.pushEntry(...)` 追加信息、成功、警告、错误事件。
- 运行时可通过 `window.__GZ_STATUS_BAR__.formatMetrics(...)` 同步底部指标卡的数字化排版。
- 运行时可通过 `window.__GZ_STATUS_BAR__.publishSimulationSummary(...)` 把仿真结果与性能趋势写入对应底部视图。
- 警告和错误事件默认进入告警视图，便于故障注入过程中的快速定位。
- 模型检查完成后会自动写入底部状态流，右侧检查面板和底部反馈形成闭环。
- 画布工具栏通过 `window.__GZ_CANVAS_COMMANDS__` 统一管理撤销、重做、视图切换、交互模式、全屏和全屏状态同步。
- 新增组件、删除组件、删除连线、保存属性、自动整理、封装子系统等结构性操作会进入 40 步撤销历史。
- 右侧属性栏移除额外提示面板，保持配置区纵向贯通和低噪声。
